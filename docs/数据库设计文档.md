# 小鲜鸡数据库设计文档

> 用大白话解释每一张表是干嘛的、每个字段什么意思。适合新加入项目的开发人员阅读。

---

## 基础概念先搞懂

### 什么是"表"？

数据库就像一个 Excel 工作簿，里面有很多**工作表**（Sheet）。一个工作表就是一张**表**。

比如「用户表」就相当于一个 Sheet，专门存所有用户的信息。

### 什么是"字段"？

字段就是 Excel 里的**列**。比如用户表有「手机号」「昵称」这些列，每一列就是一个字段。

### 什么是"数据行"？

数据行就是 Excel 里的**一行数据**。比如张三这一行记录了他的手机号、昵称，这一行就是一条数据。

### 什么是"主键"（PRIMARY KEY）？

主键就是**每行数据的唯一编号**，像身份证号一样，绝对不会重复。

比如用户表里 `id = 1` 是张三，那 `id = 1` 这辈子只属于张三，不会有第二个人用这个编号。

> **本项目的约定**：每张表都有一个 `id` 字段作为主键，类型是 `BIGINT UNSIGNED AUTO_INCREMENT`。
> 翻译：一个很大的整数，自动 +1 递增。第一条数据 id=1，第二条 id=2，以此类推。

## 逐表详解

### 1. users — 用户表（C 端顾客）

**干嘛的**：存所有买菜顾客的基本信息。顾客通过微信小程序登录，用微信的 openid 作为唯一标识。

| 字段 | 类型 | 解释 |
|------|------|------|
| `id` | BIGINT | **主键**，自增数字编号 |
| `openid` | VARCHAR(64) | 微信用户的唯一标识（每个微信号在每个小程序里有一个固定 openid，不会变） |
| `phone` | VARCHAR(20) | 手机号（顾客授权后才获取） |
| `nick_name` | VARCHAR(100) | 微信昵称 |
| `avatar_url` | VARCHAR(500) | 微信头像图片链接 |
| `role` | ENUM | 角色：`customer`（顾客）、`merchant`（商家）、`admin`（管理员）。默认 customer |
| `password` | VARCHAR(128) | 密码的加密字符串（商家端登录用，顾客不用） |
| `create_time` | DATETIME | 注册时间 |
| `last_login` | DATETIME | 最近一次登录时间 |

**大白话举例**：
> 张三打开小鲜鸡小程序，微信返回他的 openid = `oABC123xxx`。系统在 users 表里查：有这个 openid 吗？没有 → 新建一行，id 自动分配为 1。以后张三每次打开小程序，系统都通过 openid 认出他是 id=1 的张三。

| 3 | 鸡翅 | 3 |
| 4 | 鸡胸 | 4 |
| 5 | 内脏 | 5 |
| 6 | 鸽子 | 6 |

### 3. products — 商品表

**干嘛的**：存每一个商品的详细信息。这是最复杂的一张表，因为商品有三种计价方式。

| 字段 | 类型 | 解释 |
|------|------|------|
| `id` | BIGINT | **主键** |
| `name` | VARCHAR(200) | 商品名称（如"清远走地鸡"） |
| `category_id` | BIGINT | 属于哪个分类（外键，指向 categories.id） |
| `pricing_type` | ENUM | **计价方式**：`range_weight`（按重范围）、`exact_weight`（称重计费）、`per_piece`（按只卖） |
| `selling_point` | VARCHAR(200) | 卖点宣传语（如"散养180天"） |
| `description` | TEXT | 商品详情介绍 |
| `images` | JSON | 商品图片链接数组，如 `["url1.jpg","url2.jpg"]` |
| `delivery_modes` | JSON | 支持的配送方式：`["delivery"]` 可配送，`["pickup"]` 可自提 |
| `processing_options` | JSON | 加工选项：`["整只","切块"]` |
| `price_per_jin` | INT | **每斤单价**（单位：分）。比如 2500 表示 25 元/斤 |
| `weight_options` | JSON | 可选重量规格（克），如 `[500,1000,1500]` 表示半斤/一斤/一斤半 |
| `processing_fee` | INT | 切块加工费（分） |
| `unit_price` | INT | 按只卖的单价（分） |
| `specs` | JSON | 按重范围的规格配置，结构复杂，见下方说明 |
| `emoji` | VARCHAR(10) | 商品 emoji 图标，默认 🐔 |
| `sales` | INT | 销量 |
| `out_of_stock` | TINYINT | 是否售罄：0 有货，1 卖完了 |
| `status` | ENUM | `on` 上架，`off` 下架 |
| `sort` | INT | 排序 |
| `create_time` | DATETIME | 创建时间 |
| `update_time` | DATETIME | 最后更新时间 |

**三种计价方式大白话**：

| 计价方式 | 场景 | 用到哪些字段 |
|----------|------|-------------|
| `range_weight` | 鸡大小不一，按重量范围定价。比如 2-3 斤的鸡 25 元/斤，3-4 斤的鸡 22 元/斤 | `specs`（JSON 存多个规格） |
| `exact_weight` | 称重计费，只有一个单价。比如鸡腿 28 元/斤 | `price_per_jin`, `weight_options` |
| `per_piece` | 按只卖，不论重量。比如鸽子 35 元/只 | `unit_price` |

### 4. order_info — 订单主表（核心中的核心）

**干嘛的**：存每一笔订单的全部信息。从下单到支付到退款，所有状态变更都记录在这一行里。

| 字段 | 类型 | 解释 |
|------|------|------|
| `id` | BIGINT | **主键** |
| `order_no` | VARCHAR(32) | **商户订单号**（唯一）。格式如 `XJ20260711A00001` |
| `user_id` | VARCHAR(64) | 下单用户的 openid |
| `type` | ENUM | 订单类型：`delivery`（送货上门）、`pickup`（到店自提）、`offline`（线下交易） |
| `order_status` | TINYINT | 订单状态数字：0 待支付、1 已支付、2 已取消、3 已完成、4 退款中、5 已退款 |
| `status_label` | VARCHAR(20) | 状态英文标签，用于前端展示流程节点 |
| `items` | JSON | **买了什么**。数组，每个元素包含商品 ID、名称、规格、数量、单价 |
| `total_amount` | INT | 订单原价（分） |
| `pay_amount` | INT | 实付金额（分） |
| `discount_amount` | INT | 优惠金额（分） |
| `actual_amount` | INT | 称重后的实际金额（分）— 因为鸡是称重的，预估金额和实际金额可能不同 |
| `actual_weight` | INT | 实际重量（克） |
| `refund_amount` | INT | 退款金额（分） |
| `refund_status` | VARCHAR(20) | 退款状态 |
| `delivery_address` | JSON | 收货地址（姓名、手机号、省市区、详细地址、经纬度） |
| `is_scheduled` | TINYINT | 是否预约配送 |
| `scheduled_date` | VARCHAR(20) | 预约日期 |
| `scheduled_time` | VARCHAR(20) | 预约时间段 |
| `transaction_id` | VARCHAR(64) | 微信支付交易号（微信返回的） |
| `prepay_id` | VARCHAR(64) | 微信预支付 ID |
| `pay_time` | DATETIME | 支付时间 |
| `expire_time` | DATETIME | 支付超时时间（超时未支付自动取消） |
| `accept_time` ~ `complete_time` | DATETIME | 各流程节点时间（接单、称重、加工、备好、配送、完成、取消） |
| `cancel_reason` | VARCHAR(200) | 取消原因 |
| `weigh_info` | JSON | 称重详情 |
| `refund_info` | JSON | 退款详情 |
| `card_number` | VARCHAR(10) | 号码牌（自提订单用） |
| `contact_name` | VARCHAR(50) | 联系人姓名 |
| `contact_phone` | VARCHAR(20) | 联系电话 |
| `payment_type` | ENUM | 支付方式：`wechat`（微信）、`cash`（现金）、`unpaid`（未付） |
| `is_deleted` | TINYINT | 软删除标记：0 正常，1 已删除 |
| `create_time` | DATETIME | 下单时间 |
| `update_time` | DATETIME | 最后更新时间 |

**一条订单数据长什么样**（简化版）：

```
id: 128
order_no: "XJ20260711A00042"
user_id: "oABC123xxx"
type: "pickup"          ← 到店自提
order_status: 1         ← 已支付
items: [
  { productName: "清远走地鸡", spec: "2-3斤", quantity: 1, unitPrice: 2500 }
]
total_amount: 7500       ← 预估 75 元（3斤×25元/斤）
pay_amount: 7500         ← 付了 75 元
actual_amount: 6800      ← 实际称重后 68 元（2.72斤）
refund_amount: 700       ← 退了 7 元
card_number: "07"        ← 领 07 号牌
create_time: 2026-07-11 10:30:00
```

### 7. refund_alert — 退款失败告警表

**干嘛的**：退款不是每次都成功（网络问题、证书问题等）。失败了就记在这里，PC 商家端首页会显示红色角标提醒店员处理。

| 字段 | 类型 | 解释 |
|------|------|------|
| `id` | BIGINT | **主键** |
| `order_no` | VARCHAR(32) | 关联订单号 |
| `refund_no` | VARCHAR(64) | 退款单号 |
| `refund_amount` | INT | 退款金额（分） |
| `alert_type` | VARCHAR(32) | 告警类型（如 `weigh_refund_failed` 称重退款失败） |
| `error_message` | VARCHAR(500) | 失败原因（微信返回的错误信息） |
| `status` | TINYINT | 0 未处理、1 已处理、2 已忽略 |
| `resolved_by` | BIGINT | 谁处理的 |
| `resolved_at` | DATETIME | 什么时候处理的 |
| `create_time` | DATETIME | 告警产生时间 |
| `update_time` | DATETIME | 更新时间 |

### 8. stock_lock_record — 库存锁定表

**干嘛的**：顾客下单时先"占坑"，锁定库存。支付成功 → 正式扣减；支付超时 → 释放库存。

| 字段 | 类型 | 解释 |
|------|------|------|
| `id` | BIGINT | **主键** |
| `order_no` | VARCHAR(32) | 关联订单号 |
| `goods_id` | BIGINT | 商品 ID |
| `batch_no` | VARCHAR(32) | 批次号 |
| `lock_num` | INT | 锁定数量 |
| `lock_status` | TINYINT | 1 锁定中、2 已扣减、3 已释放 |
| `expire_time` | DATETIME | 锁定过期时间（超时自动释放） |
| `create_time` | DATETIME | 创建时间 |
| `update_time` | DATETIME | 更新时间 |

**大白话举例**：
> 张三下单一只 2-3 斤的清远鸡，系统从 Redis 缓存里扣掉 1 只库存，同时在 stock_lock_record 里记一条：订单 XJxxx 锁定了商品 #5，有效期 15 分钟。15 分钟内付款 → 正式扣库存。超时不付 → 库存还回去。

### 13. refresh_tokens — 登录续期凭证表

| 字段 | 类型 | 解释 |
|------|------|------|

### 14. counters — 订单号计数器表

**干嘛的**：生成唯一订单号。订单号格式是 `XJ20260711A00001`，其中 A 是字母序号，00001 是数字序号。

| 字段 | 类型 | 解释 |
|------|------|------|
| `counter_key` | VARCHAR(50) | **主键**，固定为 `order` |
| `letter_idx` | INT | 字母索引（0=A, 1=B, ... 25=Z），到 Z 后归零 |
| `seq` | INT | 数字序号（0~99999），到 99999 后归零、字母进一位 |
| `value` | INT | 通用计数器 |

### 15. merchant_accounts — 商家账号表

**干嘛的**：PC 商家端的登录账号，和顾客 users 表完全独立。

| 字段 | 类型 | 解释 |
|------|------|------|
| `id` | INT | **主键** |
| `username` | VARCHAR(50) | 登录用户名（唯一） |
| `password_hash` | VARCHAR(128) | bcrypt 加密后的密码（12 轮加密，不可逆） |
| `role` | ENUM | 角色：`admin`（管理员，最高权限）、`manager`（店长）、`staff`（店员） |
| `display_name` | VARCHAR(50) | 显示名称（如"王店长"） |
| `created_by` | INT | 谁创建的（外键，指向本表的 id。admin 是系统内置，created_by 为 NULL） |
| `is_active` | TINYINT | 是否启用：1 正常、0 禁用 |
| `last_login_at` | DATETIME | 最后一次登录时间 |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

**三种角色权限**：

| 角色 | 能干嘛 |
|------|--------|
| `admin` | 全部权限。创建/删除账号，改支付配置，看所有数据 |
| `manager` | 管理权限。创建店员账号，管理商品，处理订单 |
| `staff` | 操作权限。称重、收银、打单，不能改设置 |

**内置账号**：系统启动时自动创建 `admin` / `Admin@2024`（管理员）。

---

### 16. merchant_operation_log — 商家操作日志表

**干嘛的**：记录商家端谁的什么操作，用于审计追溯（谁在什么时候干了什么）。

| 字段 | 类型 | 解释 |
|------|------|------|
| `id` | INT | **主键** |
| `operator_id` | INT | 操作者 ID（merchant_accounts.id） |
| `operator_name` | VARCHAR(50) | 操作者名字（冗余，方便直接看不用 JOIN） |
| `action` | VARCHAR(50) | 操作类型 |
| `target_id` | INT | 被操作的对象 ID（如被修改的账号 ID） |
| `target_name` | VARCHAR(50) | 被操作的对象名称 |
| `detail` | JSON | 操作详情（如修改了什么字段、旧值新值） |
| `ip` | VARCHAR(45) | 操作者 IP 地址 |
| `created_at` | DATETIME | 操作时间 |

**记录哪些操作**：

| action 值 | 场景举例 |
|-----------|---------|
| `create_account` | 管理员创建了一个新店员账号"小李" |
| `delete_account` | 管理员删除了离职员工的账号 |
| `reset_password` | 管理员帮店员重置密码 |
| `toggle_active` | 管理员禁用/启用了某个账号 |
| `update_account` | 管理员修改了店员信息 |
| `change_own_password` | 店员自己修改了密码 |

---

### 17. payment_methods — 支付方式配置表

**干嘛的**：PC 商家端可以新建/编辑/删除支付方式。支持微信支付和支付宝，每套配置可以有不同的商户号。

| 字段 | 类型 | 解释 |
|------|------|------|
| `id` | INT | **主键** |
| `name` | VARCHAR(30) | 支付名称（如"微信支付主商户"） |
| `channel` | ENUM | 渠道：`wechat`（微信支付）、`alipay`（支付宝） |
| `merchant_type` | ENUM | 商户类型：`normal`（普通商户）、`service`（服务商） |
| `app_id` | VARCHAR(32) | 微信 AppID |
| `app_secret` | VARCHAR(64) | 微信 AppSecret |
| `mchid` | VARCHAR(10) | 微信支付商户号（10 位数字） |
| `serial_no` | VARCHAR(40) | APIv3 证书序列号 |
| `api_key` | VARCHAR(64) | APIv3 密钥（32 位） |
| `cert_pem` | TEXT | 商户证书内容 |
| `key_pem` | TEXT | 商户私钥内容 |
| `alipay_app_id` | VARCHAR(32) | 支付宝 AppID |
| `alipay_private_key` | TEXT | 支付宝应用私钥 |
| `alipay_public_key` | TEXT | 支付宝公钥 |
| `enabled` | TINYINT | 是否启用：1 启用、0 禁用 |
| `is_default` | TINYINT | 是否默认：1 是、0 否 |
| `created_by` | INT | 谁创建的（merchant_accounts.id） |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

**安全设计**：`api_key`、`cert_pem`、`key_pem`、`alipay_private_key` 等敏感字段在列表接口中脱敏显示为 `***`，只在编辑时通过单独接口获取完整值。

## 表之间的关系

```
顾客端（C 端）流程：
  users (顾客)
    ↓ 浏览
  categories → products (分类→商品)
    ↓ 下单
  order_info (订单) ← pai_numbers (号码牌)
    ↓ 支付
  payment_record (支付流水) ← payment_methods (支付配置)
    ↓ 退款
  refund_record (退款流水) → refund_alert (退款失败告警)
    ↓ 库存
  stock_lock_record (库存锁定)

商家端（B 端）流程：
  merchant_accounts (商家账号)
    ↓ 操作日志
  merchant_operation_log (操作审计)
```
